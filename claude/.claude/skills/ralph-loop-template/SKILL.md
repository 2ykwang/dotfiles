## Instructions

### Step 1: 계획 수집

**인자가 제공된 경우**: 해당 파일을 읽어 계획 내용을 파악한다.

**인자가 없는 경우**: 현재 대화 맥락에서 논의된 계획을 사용한다.
대화 맥락에도 계획이 없으면 사용자에게 계획을 설명해달라고 요청한다.

### Step 2: 계획 분석

계획에서 다음을 추출한다:

- **목표**: 무엇을 만드는가
- **비목표**: 이 작업에서 하지 않을 것 (Feature Invention 방지용)
- **페이즈 목록**: 순서가 있는 구현 단계들
- **각 페이즈의 핵심 내용**: 변경 파일, 구현 내용, 검증 방법
- **참조 문서/파일**: 패턴 참조, 설계 문서 경로
- **완료 조건**: 전체 작업이 끝났다고 판단하는 기준 (객관적으로 측정 가능해야 함)

### Step 3: 페이즈 분할

계획을 Ralph Loop 이터레이션에 적합한 페이즈로 분할한다.

**페이즈 분할 원칙**:

- **한 이터레이션 = 정확히 한 페이즈만 처리한다**. 2개 이상은 절대 금지.
- 한 페이즈가 완료되면 즉시 이터레이션을 종료한다. 다음 페이즈는 다음 이터레이션에서 처리한다.
- 모든 페이즈가 완료될 때까지 이터레이션은 자동으로 반복된다.
- 각 페이즈는 **독립적으로 검증 가능한 단위**여야 한다 (빌드 통과, 테스트 통과 등).
- 너무 크면 에이전트가 중간에 품질을 놓친다. 너무 작으면 이터레이션 낭비다.
- **적절한 크기 기준**: 파일 1~3개 수정, 코드 50~150줄 변경 정도.
- 원본 계획의 단계가 너무 크면 쪼개고, 너무 작으면 합친다.

### Step 4: PROMPT 파일 생성

프로젝트 루트에 `PROMPT-{kebab-case-이름}.md` 파일을 생성한다.

파일 구조:

```markdown
# {목표 제목} — Ralph Loop Prompt

당신은 {프로젝트명}에 {목표}를 구현하는 에이전트입니다.
이 프롬프트는 매 반복마다 동일하게 전달됩니다.
파일이 변경되어 있으므로 항상 현재 상태를 먼저 확인하세요.

---

## 절대 규칙

1. **한 이터레이션 = 정확히 한 페이즈만 처리한다**.
   - 한 페이즈를 완료했으면 나머지 페이즈가 남아 있어도 즉시 이터레이션을 종료한다.
   - 다음 페이즈는 다음 이터레이션에서 처리된다. 이터레이션은 자동으로 반복된다.
   - "효율을 위해" 2개를 한 번에 처리하는 것을 절대 하지 않는다.
2. **이미 구현된 코드를 가정하지 않는다**: 항상 파일을 읽어서 현재 상태를 확인한다.
3. **명세에 없는 기능을 추가하지 않는다**: 체크리스트와 참조 문서에 명시된 것만 구현한다.
4. {프로젝트별 규칙들 - CLAUDE.md에서 핵심 규칙 추출}

## 매 이터레이션 절차

> **원칙**: 한 이터레이션은 반드시 하나의 페이즈만 처리한다.
> 페이즈가 여러 개 남아 있어도 하나만 완료하고 종료한다.
> 이터레이션이 종료되면 루프가 다음 이터레이션으로 넘어간다.
> 모든 페이즈가 완료될 때까지 이 사이클이 반복된다.
> **루프를 임의로 멈추지 않는다** — promise 출력만이 루프를 종료하는 유일한 수단이다.

### STEP 1: 현재 상태 파악
이 프롬프트 파일을 읽어 체크리스트의 현재 상태를 확인한다.
첫 번째로 미완료 항목(`[ ]`)이 남아있는 페이즈를 찾는다.
→ **이 페이즈 하나만 이번 이터레이션에서 처리한다.**

### STEP 2: 명세 확인
참조 문서에서 해당 페이즈의 상세 내용을 읽는다.
관련 기존 코드도 읽어서 패턴을 파악한다.

### STEP 3: 구현
해당 페이즈의 서브 항목들을 순서대로 구현한다.
완료된 서브 항목은 즉시 `[x]`로 체크한다.
**다른 페이즈에 손대지 않는다.**

### STEP 4: 검증
{검증 명령어}. 실패하면 고친다.
**검증을 통과하지 못하면 promise를 출력하지 않는다.**

### STEP 5: 이터레이션 종료 및 다음 이터레이션 시작
페이즈 내 모든 서브 항목이 `[x]`이면 이번 이터레이션을 종료한다.

**미완료 페이즈가 남아 있는 경우:**
- 이터레이션을 종료하면 루프가 다음 이터레이션으로 넘어간다.
- 아무 말 없이 조용히 종료한다. 루프가 알아서 이 프롬프트를 다시 전달한다.
- "다음에 계속하겠습니다" 같은 안내를 출력하지 않는다. 그냥 종료한다.

**모든 페이즈가 `[x]`인 경우:**
- 완료 조건을 확인하고 promise를 출력한다. 루프가 종료된다.

## 트러블슈팅

이터레이션이 실패하면 이 프롬프트를 튜닝한다. 에이전트의 실패는 예측 가능하고 수정 가능하다.

- **빌드 실패 반복**: 참조 문서의 코드 예시를 더 구체적으로 보강
- **범위 벗어남**: 비목표 섹션에 제한 사항 추가
- **품질 저하**: 페이즈를 더 작은 단위로 분할
- **무한 루프**: 검증 조건이 달성 불가능하지 않은지 확인
```

### 체크리스트 작성 규칙

- **페이즈 = 이터레이션 단위**: 하나의 `### Phase N` 블록이 하나의 이터레이션에서 처리된다
- **서브 항목 = 페이즈 내 세부 작업**: 각 Phase 아래에 2~5개의 `- [ ]` 서브 항목을 둔다
- 서브 항목에는 변경 파일과 구현 내용을 명시한다
- 각 Phase의 마지막 서브 항목은 검증 단계 (e.g., "검증: `go build ./...`")
- 마지막 Phase는 항상 전체 검증 (테스트 + 빌드)

### 비목표 작성 규칙

계획에서 명시적으로 제외된 사항 외에도, 다음을 추론하여 추가한다:
- 계획의 범위를 벗어나는 리팩토링
- 요청되지 않은 테스트/문서/설정 변경
- 기존 API나 인터페이스 변경 (계획에 명시되지 않은 경우)

### COMPLETION_PROMISE 생성 규칙

- 대문자 영문, 공백 구분
- 목표를 짧게 요약 (e.g., "HELP OVERLAY COMPLETE", "AUTH REFACTOR DONE")

### Step 5: 결과 출력

생성된 PROMPT 파일 경로를 알려주고, 아래 형식으로 `/ralph-loop` 명령어를 제공한다:

```
### 생성된 파일

`{PROMPT 파일 경로}`

### Ralph Loop 실행 명령어

/ralph-loop:ralph-loop "Read {PROMPT 파일 경로} and implement the next unchecked phase. Always read the file first to find the first phase with [ ] items." --max-iterations {Phase 수 + 2} --completion-promise "{COMPLETION_PROMISE}"
```

위 명령어 블록을 그대로 복사하여 사용할 수 있도록 출력한다.
